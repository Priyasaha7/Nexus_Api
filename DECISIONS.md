# NexusAPI — Decision Log

## Question 1: Why does the credit system use a transaction ledger instead of a balance column?

The credit system uses a ledger of transactions rather than a single balance column because a ledger is fundamentally safer and more correct for financial data. If the system stored only a balance column, two problems would emerge immediately. First, there would be no audit trail — if a user disputed a charge, there would be no way to prove when credits were added or deducted and why. Second, and more critically, a balance column creates a race condition: two simultaneous requests could both read the same balance, both decide they have enough credits, and both subtract from it, resulting in a negative balance or a double-spend. The ledger approach solves both problems. Every credit movement — whether a grant or a deduction — is written as a new row with an amount, reason, timestamp, and user reference. The current balance is always derived by summing these rows with a SQL SUM() query. This means the balance is always mathematically correct and fully auditable. The tradeoff is that reading the balance requires a SUM query rather than a simple column read, which is slightly slower. However, this cost is negligible at the scale this system operates, and the correctness guarantees are worth it. This is the same design used by Stripe, banks, and every serious payment system.

## Question 2: How did you handle the simultaneous credit deduction problem?

The simultaneous deduction problem — where two requests arrive at exactly the same time with exactly enough credits for only one — is handled using PostgreSQL's SELECT FOR UPDATE mechanism. When deduct_credits is called, before checking the balance or inserting a transaction, the query that reads the current balance includes a .with_for_update() clause. This tells PostgreSQL to acquire a row-level lock on the credit transaction rows for that organisation. The first request to arrive acquires the lock and proceeds: it reads the balance, confirms it is sufficient, and inserts the deduction row. The second request, arriving simultaneously, is forced to wait at the lock. When the first request commits and releases the lock, the second request reads the now-updated balance, finds it insufficient, and raises an InsufficientCreditsError. This means only one of the two requests succeeds. I chose this approach over application-level checks because database locks are atomic — the check and the deduction happen inside a single transaction, with no gap where a second request could slip through. An alternative would have been optimistic locking with a version column, but that requires retry logic and is more complex to implement correctly.

## Question 3: What happens when the background worker fails after credits have been deducted?

When a summarise job is queued, credits are deducted immediately and a job record is created with status pending. If the ARQ worker crashes before processing the job, the credits remain deducted and the job stays in pending or failed state. I made a deliberate decision not to implement automatic refunds for failed jobs. The reason is that automatic refunds create their own race conditions and complexity — a refund system needs to be idempotent itself, and implementing it correctly is a significant engineering problem. Instead, the system marks the job as failed with an error message, and the job record is permanently stored so an administrator can identify and manually refund affected organisations. In a production system, I would add a scheduled task that scans for jobs that have been in pending state for more than five minutes and marks them as failed, then adds them to a refund queue. The five-minute threshold was chosen because it is long enough for any legitimate slow job to complete, but short enough to catch stuck jobs before users notice.

## Question 4: How does your idempotency implementation work, and where does it live?

Idempotency is implemented at two levels. At the application level, when a request arrives with an Idempotency-Key header, the system first checks the idempotency_records table for an existing record with that key belonging to the same organisation within the last 24 hours. If a record is found, the stored response is returned immediately without touching credits or doing any work. If no record is found, the request is processed normally, and after a successful response, the response body is saved to idempotency_records for future duplicate requests. At the database level, the credit_transactions table has a unique constraint on the idempotency_key column. This means even if two identical requests arrive simultaneously — before either has completed — only one can succeed in inserting a credit transaction with that key. The second will receive a database IntegrityError, which the application catches and handles by fetching and returning the original transaction. This two-layer approach means idempotency is enforced both in application logic for the common case and at the database constraint level for the concurrent case. The database constraint is the true safety guarantee — the application-level check is an optimisation that avoids unnecessary database work.

## Question 5: What would break first at 10x the current load, and what would you do about it?

At ten times the current load, the first bottleneck would be the PostgreSQL database, specifically the SUM query used to calculate the credit balance on every API call. This query scans all credit transaction rows for an organisation every time a balance check is needed. At low volume this is fast, but as transaction history grows and request volume increases, this query becomes the slowest part of every request. The fix I would implement is a materialised balance cache: a separate column on the organisations table that stores the current balance and is updated atomically inside the same database transaction as each credit deduction or grant. This turns the balance check from a full table scan into a single indexed column read. The risk with this approach is that the cached balance must always be updated inside the same transaction as the transaction row insertion, otherwise the two can drift apart. A second bottleneck at 10x load would be the Redis rate limiter, which receives one read and one write per request. This could be addressed by switching from a fixed-window to a sliding-window algorithm and using Redis cluster mode for horizontal scaling.
